## 1. Purpose
This document describes the Information Security requirements for maintaining up-to-date Security patches on all systems.

## 2. Objective
- The objective of this policy is to prevent the exploitation of vulnerabilities on computing and related systems.
- All Systems, Servers, and Network Devices must be maintained, and security patches must be applied in a timely manner consistent with an assessment of risk.

## 3. Scope
This policy applies to all Systems, Network Devices, and Servers of Trianz which are on-premise or any Cloud-based infrastructure or service where Patch Management is Trianzâ€™s responsibility. This includes systems that contain company or client data owned or managed by Trianz, regardless of location (On-prem or Cloud).

## 4. Policy Statement

### 4.1 Trianz Owned Assets
- Systems, Servers, and infrastructure owned and managed by Trianz must have the latest security patches installed to protect against known and unknown vulnerabilities.
- Security patches must be identified, reviewed, tested, and approved before implementation in the production environment.
- All security patch updates must follow the Change control process to ensure successful completion.

### 4.2 Supplier Owned Assets
- Systems, Servers, and infrastructure owned by the Supplier and managed by Trianz/Supplier must have the latest security patches installed to protect against known and unknown vulnerabilities.
- If patches are to be applied from the supplier's end, the contract must clearly specify this requirement.
- Security patches must be identified, reviewed, tested, and approved before implementation in the production environment.
- All security patch updates must follow the Change control process to ensure successful completion.
- Trianz's supplier security policy must enforce security patch compliance by Suppliers/vendors through periodic vendor audits and SLA reviews.

## 5. Patching Schedule

### IT Infrastructure
- **Monthly**: Apply critical security patches, security updates, general system updates, and critical third-party application patches and upgrades based on security recommendations.
- **Quarterly**: Perform server updates.
- **On-demand**: Deploy high/critical severity patches immediately based on security recommendations.

### Printers, Peripherals, Switches, Routers, Firewalls, and Storage
- **Quarterly**: Check for new firmware updates using OEM portals, emails, security blogs, etc.
- **Vendor Update**: Upgrade firmware within a week of vendor notifications.

#### 5.1.1.1 On-demand and Emergency Security Patching
- The IS Security team will determine the risk and relevance of the patch, as well as when the system should be patched. All related critical, high, and business impact patches must be deployed immediately upon approval.

#### 5.1.1.2 Zero-day Vulnerabilities
- **Immediate Risk Assessment**: Identify affected systems, assess impact, and determine exposure.
- **Emergency Patch Deployment**: Apply vendor-released patches or hotfixes as soon as they become available.
- **Temporary Mitigations**: If no patch is available, implement workarounds such as disabling affected services, applying firewall rules, or restricting access.
- **Expedited Testing**: Conduct rapid testing of patches in a controlled environment before deployment to prevent system disruptions.
- **Prioritized Deployment**: Apply patches on critical systems first, followed by non-critical assets.
- **Monitoring & Threat Intelligence**: Continuously monitor for new threats, indicators of compromise, and vendor updates.
- **Incident Response Integration**: Align with the incident response process to contain, eradicate, and recover from potential exploitation.
- **Post-Patch Review**: Validate successful updating of patches, conduct security testing, and update documentation for future reference.

### 5. Exceptions
- Exceptions to the patch management policy must be documented by the device/application owner, including justification, and necessary compensating controls must be implemented for systems, network devices, and security appliances that cannot be patched to resolve known vulnerabilities.

### 6. Monitoring and Reporting
- The IS Operation team shall generate and maintain patch implementation reports to summarize the outcome of each patching cycle.
- These reports will be used to evaluate patch implementation, current patching levels of all systems, and to assess the current level of risk.

### 7. Roles and Responsibilities
- **System Owner**: Responsible for ensuring that the software they manage is maintained through regular software updates and patching.
- **CIO**: Accountable for ensuring adherence to the software update and patching policy.
- **IS Operation**: Responsible for the implementation and enforcement of this policy.
- **InfoSec Team**: Conducts periodic/spot audits to ensure compliance.
- **Product Owner/Project Manager**: Ensures all endpoint systems used for projects are patched.
