## 1. Purpose
This document describes the Information Security requirements for maintaining up-to-date Security patches on all systems.

## 2. Objective
- The objective of this policy is to prevent the exploitation of vulnerabilities on computing and related systems.
- All Systems, Servers, and Network Devices must be maintained, and security patches must be applied in a timely manner consistent with an assessment of risk.

## 3. Scope
This policy applies to all Systems, Network Devices, and Servers of Trianz which are on-premise or any Cloud-based infrastructure or service where Patch Management is Trianzâ€™s responsibility. This includes systems that contain company or client data owned or managed by Trianz regardless of location (On-prem or Cloud).

## 4. Policy Statement

### 4.1 Trianz Owned Assets
- Systems, Servers, and infrastructure owned and managed by Trianz must be installed with the latest security patches to protect against known and unknown vulnerabilities.
- Security patches must be identified, reviewed, tested, and approved before implementation in the production environment.
- All security patch updates must follow the Change Control process to ensure successful completion.

### 4.2 Supplier Owned Assets
- Systems, Servers, and infrastructure owned by the Supplier and managed by Trianz/Supplier must be installed with the latest security patches to protect against known and unknown vulnerabilities.
- Contracts must clearly specify the application of patches from the supplier end.
- Security patches must be identified, reviewed, tested, and approved before implementation in the production environment.
- All security patch updates must follow the Change Control process to ensure successful completion.
- Trianz's supplier security policy must enforce security patch compliance by Suppliers/vendors through periodic vendor audits and SLA reviews.

## 5. Patching Schedule

### IT Infrastructure
- **Monthly**: Apply critical security patches, security updates, and general system updates, including critical third-party application patches and upgrades based on security recommendations.
- **Quarterly**: Perform server updates.
- **On-demand**: Deploy high/critical severity patches immediately based on security recommendations.

### Printers, Peripherals, Switches, Routers, Firewalls, and Storage
- **Quarterly**: Check for new firmware updates by referencing OEM portals, emails, security blogs, etc.
- **Vendor Update**: Upgrade firmware within a week of vendor notifications, referencing vendor notifications.

**5.1.1.2 Zero-day Vulnerabilities**

- **Immediate Risk Assessment**: The IS Security team will identify affected systems, assess the impact, and determine exposure.
- **Emergency Patch Deployment**: Vendor-released patches or hotfixes must be applied as soon as they become available.
- **Temporary Mitigations**: If no patch is available, implement workarounds such as:
  - Disabling affected services
  - Applying firewall rules
  - Restricting access
- **Expedited Testing**: Conduct rapid testing of patches in a controlled environment before deployment to prevent system disruptions.
- **Prioritized Deployment**: Patches should be applied to critical systems first, followed by non-critical assets.
- **Monitoring & Threat Intelligence**: Continuously monitor for new threats, indicators of compromise, and vendor updates.
- **Incident Response Integration**: Align patching efforts with the incident response process to contain, eradicate, and recover from potential exploitation.
- **Post-Patch Review**: Validate successful patch updates, conduct security testing, and update documentation for future reference.

**Exceptions**: Systems, network devices, security appliances, and applications that cannot be patched to resolve known vulnerabilities must have justification documented by the device/application owner, and necessary compensating controls must be implemented.

**Systems**

- All systems (desktops and laptops) must have automatic updates enabled for security patches for the operating system.
- All endpoints will be patched for critical third-party applications or new version upgrades based on security recommendations.

**Servers and Network/Security Devices**

- Servers (Unix, Linux, Windows, etc.) must comply with operating system patches released by vendors or as per the Trianz patch update schedule, whichever is earlier.
- Network/security devices must comply with internet operating systems, firmware, and patches released by OEMs and vendors or as per the Trianz patch update schedule, whichever is earlier.
- The above patch schedule is defined to address the default operating system level, service pack, hotfix, and patch level required to ensure the security of Trianz assets and the data residing on the system.

**6. Monitoring and Reporting**

- Patch implementation reports shall be generated and maintained by the IS Operation team to summarize the outcome of each patching cycle.
- These reports will be used to evaluate patch implementation, assess the current patching levels of all systems, and determine the current level of risk.

**7. Roles and Responsibilities**

- **Role**: System Owner  
  **Responsibility**: Ensure that the software they manage is maintained.  
  **Internal/External**: Internal

### 7. Responsibilities

- **CIO**: The Chief Information Officer (CIO) is accountable for ensuring adherence to the software update and patching policy.
  
- **IS Operation**: The IS Operation team is responsible for the implementation and enforcement of the software update and patching policy.

- **InfoSec Team**: The Information Security Team will conduct periodic and spot audits to ensure compliance with the policy.

- **Product Owner / Project Manager**: The Product Owner or Project Manager must ensure that all endpoint systems used for projects are patched.
